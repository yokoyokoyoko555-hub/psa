# SUBMISSION_BOOKING — カード提出予約 リファイン設計書

> Codex / Claude Code 共有用の自己完結ドキュメント。関連: [ADR-0014](DECISIONS.md) / 規約は [AGENTS.md](../AGENTS.md)
> 状態: **実装済（一部方針変更あり、下記参照）** / 最終更新: 2026-07-07
>
> **2026-07-07 方針変更**: §5「店頭提示レシート」のうち**カード明細リストによる現物照合は廃止**。店頭受付は**受付番号（申込番号）の画面提示＋本人確認書類での本人確認のみ**とする（[ADR-0040](DECISIONS.md)）。カードの内容確認はスタッフが別途（受付後）行う。§1〜§4・§6〜§9の予約リスト化・日時/方法選択・代理申込のお預け予約に関する設計は変更なし。

## 1. 背景・目的
支払い完了後の「カード提出予約」を、**カレンダー起点 → 申込（Application）起点のリスト型**へ作り替える。
あわせて、(a) 予約完了後は専用の詳細ページへ遷移、(b) 既存予約の管理を一覧で、(c) 店頭持込時に**店員と現物を面前照合するためのレシート（カード明細リスト）**を提供する。

> 予約は既存スキーマで **申込ごとに1件**（`SubmissionBooking.applicationId @unique`）。本リファインは主にUI/動線の変更で、スキーマ変更は原則不要。

## 2. 確定方針（2026-06-23、ユーザー承認済み）
| 論点 | 採用 |
|------|------|
| 予約トップ | **支払済み申込のリスト**（申込番号・枚数・金額・予約状態）。カレンダーは選択した申込専用に従属表示 |
| 予約完了後 | 専用の**予約詳細ページ**へ遷移（完了画面を兼ねる） |
| 予約管理 | 上記リスト＋詳細ページで変更・キャンセル |
| 店頭提示ページ | **QRなしのレシート**。**カード明細リストを表示し、店員が現物と1枚ずつ面前照合**する用途（QRだと現物照合にならないため不採用） |
| 進め方 | 本設計書＋ADR-0014 を先に記録 → 実装 |

## 3. 画面・ルート
| ルート | 種別 | 内容 |
|--------|------|------|
| `/mypage/submission-booking` | 改修 | **支払済み申込のリスト**。各行: 申込番号 / カード枚数 / 金額 / 予約状態バッジ（未予約 or 予約済[日時・方法]）。行アクション: 「予約する」/「予約を見る・変更」 |
| `/mypage/submission-booking/[applicationId]` | 新規 | **予約詳細＝完了ページ＝店頭提示レシート**（§5）。「変更」「キャンセル」 |
| `/mypage/submission-booking/[applicationId]/edit` | 新規(client) | 単一申込の**日時・方法選択**（休業日/発送日を反映した簡易カレンダー）。保存→詳細へ遷移 |

> 既存の `BookingCalendar`（申込選択を内包した1画面）は、申込選択を外して「単一申込の日時選択」に簡素化、または edit 用に新コンポーネント化。

## 4. 動線
```
申込決済完了
  → 予約リスト（該当申込が「未予約」で強調表示）
  → 「予約する」 → /[applicationId]/edit（方法=店頭/郵送・日時選択）
  → 保存 → /[applicationId]（予約詳細＝完了＝受付番号ページ）
  → 店頭ではこの詳細ページ（受付番号）を提示し、本人確認のみで受付
```

## 5. 受付番号ページ（/[applicationId]）の要件 ※2026-07-07改訂（ADR-0040）
~~店員と顧客が現物カードとリストを面前で突合~~ → **店頭受付は受付番号の提示＋本人確認のみに簡略化**。カードの内容確認（現物照合）は受付後にスタッフが別途行う運用に変更。
- 申込番号（大きく表示。**受付番号として利用**）／予約日時／提出方法（店頭持込 or 郵送）
- 顧客名（暗号化→復号表示。本人確認書類との照合用）
- 注意書き「店頭ではこの受付番号の画面提示と本人確認書類でのご本人確認のみを行います」
- ~~カード明細リスト（照合用）~~ → **廃止**（現物照合はこのページでは行わない）
- QRは載せない（現状は申込番号の目視提示のみ。QR化する場合は別途ADR）

## 6. データ／アクションへの影響
- **スキーマ変更なし**（`SubmissionBooking` は申込単位で既存。カード明細は `Card` から取得）。
- Server Actions: 既存を流用
  - `upsertSubmissionBooking({ applicationId, method, scheduledAt, note })`（保存）
  - `cancelSubmissionBooking(...)`（キャンセル）
  - 一覧・詳細は read（`prisma.application` + `submissionBooking` + `cards`）
- 管理側: 変更なし。店員は管理画面で申込番号検索→受領（`RECEIVED_BY_STORE`）。**QRスキャン受領は本スコープ外**（将来検討する場合は新ADR）。

## 7. 段階リリース
- **Phase 1**: 予約トップのリスト化＋ `/[applicationId]/edit`（単一申込の日時/方法選択）＋保存後に詳細へ遷移
- **Phase 2**: `/[applicationId]` 予約詳細を**店頭提示レシート**として整備（カード明細リスト・照合用体裁・提出方法別案内・印刷対応）
- 将来（別ADR）: QRスキャンによる店頭受領の効率化 など

## 8. 代理申込（source=STORE）のお預け予約 ※支払い前に必要
代理申込は「依頼 → **顧客がカードをお預け（店頭/郵送）** → スタッフが明細入力・料金確定 → 支払い」の順。
よって**お預け予約は支払い前に行う**。通常申込（CUSTOMER）の「支払い済みのみ予約可」とは条件が異なる。

- 予約対象の判定（リスト/edit/Action）:
  - `source=STORE` … `status != CANCELLED` なら**支払い前でも予約可**
  - `source=CUSTOMER` … 従来どおり `status not in (DRAFT,CANCELLED)` かつ 決済成功
- 代理申込は作成時 `status=DRAFT`／カード明細なし。リスト/レシートでは「代理申込（スタッフが明細入力）」と表示し、レシートのカード明細は「お預け後に入力」と案内。
- 代理申込の依頼完了画面（`StoreRequestForm`）から **`/mypage/submission-booking/[applicationId]/edit`** へ誘導（`createStoreRequest` が `applicationId` を返す）。

## 9. 留意点
- 認可: 自分の申込/予約のみ閲覧・編集（[ADR-0005] に倣いページ/Action側で `customerId` 一致を確認）。
- 予約可能日: `SubmissionCalendarDay`（休業日 `isClosed` / 発送日 `isShippingDay`）を edit のカレンダーに反映（既存ロジック流用）。
- 既存挙動（料金・ステータス17段階・通常申込の決済ゲート）は変更しない。代理申込の予約解放のみ `source=STORE` に限定して追加（[AGENTS.md §7]、ユーザー承認済み）。
